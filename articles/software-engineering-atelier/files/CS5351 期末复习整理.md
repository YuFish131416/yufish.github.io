>

>
>     *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT上下文，产品待办列表管理）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">The product owner shows up at the sprint planning meeting with the prioritized agile product backlog and describes the top items to the team.</span>

>     *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">产品负责人带着已确定优先级的敏捷产品待办列表出席冲刺计划会议，并向团队描述最高优先级的条目。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">核心职责</span>**<span style="color: rgb(15, 17, 21)">：管理和优先排序产品待办列表，代表客户和利益相关者。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Scrum Master / 团队 | Scrum Master / Team</span>**
>
>     *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，团队协作）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">The team then determines which items they can complete during the coming sprint. The team then moves items from the product backlog to the sprint backlog.</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">然后团队确定他们在接下来的冲刺中可以完成哪些条目。随后，团队将条目从产品待办列表移至冲刺待办列表。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">核心职责</span>**<span style="color: rgb(15, 17, 21)">：自组织地完成冲刺目标，Scrum Master负责移除障碍、确保过程顺利。</span>

>     *   **<span style="color: rgb(15, 17, 21)">解释</span>**<span style="color: rgb(15, 17, 21)">：Sprint是Scrum的核心迭代周期，通常为2-4周，期间团队完成一组已承诺的工作。</span>

>     *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，Scrum过程描述）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">driven by daily and periodic one-hour meetings... >
>     *   **<span style="color: rgb(15, 17, 21)">区别</span>**<span style="color: rgb(15, 17, 21)">：评审（Review）关注</span>**<span style="color: rgb(15, 17, 21)">产品增量</span>**<span style="color: rgb(15, 17, 21)">（向利益相关者展示成果）；回顾（Retrospective）关注</span>**<span style="color: rgb(15, 17, 21)">改进过程</span>**<span style="color: rgb(15, 17, 21)">（团队内部反思）。</span>

> ### **<span style="color: rgb(15, 17, 21)">📘 用户故事 | User Story</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT P55）</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">As a </span>**<span style="color: rgb(15, 17, 21)">&#x3C;role></span>**<span style="color: rgb(15, 17, 21)">, I want an </span>**<span style="color: rgb(15, 17, 21)">&#x3C;action></span>**<span style="color: rgb(15, 17, 21)">, so that I can achieve a </span>**<span style="color: rgb(15, 17, 21)">&#x3C;goal></span>**<span style="color: rgb(15, 17, 21)">.</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">A user story usually has more information such as use cases, diagram sketch, user notes, data, reports from existing systems to clarify the context and scope of the user story.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">作为</span>**<span style="color: rgb(15, 17, 21)">&#x3C;角色></span>**<span style="color: rgb(15, 17, 21)">，我想要</span>**<span style="color: rgb(15, 17, 21)">&#x3C;动作></span>**<span style="color: rgb(15, 17, 21)">，以便我能实现</span>**<span style="color: rgb(15, 17, 21)">&#x3C;目标></span>**<span style="color: rgb(15, 17, 21)">。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">一个用户故事通常有更多的信息，例如用例、图示草图、用户笔记、数据、现有系统的报告，以阐明用户故事的上下文和范围。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">要素</span>**<span style="color: rgb(15, 17, 21)">：</span>**<span style="color: rgb(15, 17, 21)">角色</span>**<span style="color: rgb(15, 17, 21)">（需求工程中识别）、</span>**<span style="color: rgb(15, 17, 21)">动作</span>**<span style="color: rgb(15, 17, 21)">（应是具体的、可操作的）、</span>**<span style="color: rgb(15, 17, 21)">目标</span>**<span style="color: rgb(15, 17, 21)">（应是可衡量和量化的）。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 相对故事点估算 | Relative Story Point Estimation</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT P56）</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">A story point is a value.</span>
>     > *   <span style="color: rgb(15, 17, 21)">Pick one or more user stories as anchors. The whole team agreed on a story point for each such user story to indicate the number of expected efforts to implement the user stories, taking into account uncertainty and implementation complexity.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">故事点是一种价值。</span>
>     > *   <span style="color: rgb(15, 17, 21)">选择一个或多个用户故事作为锚点。整个团队为每个这样的用户故事商定一个故事点，以表示实现这些用户故事的预期工作量，同时考虑到不确定性和实现的复杂性。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">关键</span>**<span style="color: rgb(15, 17, 21)">：是</span>**<span style="color: rgb(15, 17, 21)">相对估算</span>**<span style="color: rgb(15, 17, 21)">（相对于锚点故事），而非绝对时间；考虑的是</span>**<span style="color: rgb(15, 17, 21)">工作量、复杂性和不确定性</span>**<span style="color: rgb(15, 17, 21)">的综合。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 项目速度 | Project Velocity</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT P57, P58）</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">A user story is implemented in a cycle if it passes the cycle review (by users and product owner).</span>
>     > *   <span style="color: rgb(15, 17, 21)">Each cycle implements a set of user stories with assigned story points.</span>
>     > *   <span style="color: rgb(15, 17, 21)">The total story points delivered by the cycle are plotted.</span>
>     > *   **<span style="color: rgb(15, 17, 21)">Project velocity is the slope over a consecutive series of cycles.</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">如果用户故事通过了周期评审（由用户和产品负责人评审），则该用户故事在一个周期内完成。</span>
>     > *   <span style="color: rgb(15, 17, 21)">每个周期实现一组已分配故事点的用户故事。</span>
>     > *   <span style="color: rgb(15, 17, 21)">绘制出该周期所交付的总故事点数。</span>
>     > *   **<span style="color: rgb(15, 17, 21)">项目速度表现为连续一系列周期的斜率。</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">作用</span>**<span style="color: rgb(15, 17, 21)">：用于衡量团队在单个冲刺中交付价值的平均能力，是未来冲刺计划的关键预测指标。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 燃尽图 | Burndown Chart</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT P58，结合发布燃尽图描述）</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">release burndown chart</span>**<span style="color: rgb(15, 17, 21)"> tracks progress on a project. The chart itself is updated after each sprint. Teams can measure progress in any unit they choose.<br>Project velocity is the slope over a consecutive series of cycles.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">发布燃尽图</span>**<span style="color: rgb(15, 17, 21)">用于跟踪项目进度。该图表会在每次冲刺后更新。团队可以选择任意单位来衡量进度。<br>项目速度表现为连续一系列周期的斜率。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">解读</span>**<span style="color: rgb(15, 17, 21)">：</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">X轴</span>**<span style="color: rgb(15, 17, 21)">：时间（冲刺周期）。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">Y轴</span>**<span style="color: rgb(15, 17, 21)">：剩余工作（总故事点或任务数）。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">理想线</span>**<span style="color: rgb(15, 17, 21)">：一条从冲刺开始的总工作量到冲刺结束为零的直线。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">实际线</span>**<span style="color: rgb(15, 17, 21)">：团队实际完成工作的趋势。如果实际线在理想线之上，说明进度滞后；反之则超前。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">目的</span>**<span style="color: rgb(15, 17, 21)">：提供项目进度和趋势的视觉化反馈，便于及时调整。</span>

>
> *   **<span style="color: rgb(15, 17, 21)">RE</span>**<span style="color: rgb(15, 17, 21)"> is a process to </span>***<span style="color: rgb(15, 17, 21)">find out</span>***<span style="color: rgb(15, 17, 21)"> and </span>***<span style="color: rgb(15, 17, 21)">structure</span>***<span style="color: rgb(15, 17, 21)"> the </span>**<span style="color: rgb(15, 17, 21)">functional</span>**<span style="color: rgb(15, 17, 21)"> and </span>**<span style="color: rgb(15, 17, 21)">non-functional</span>**<span style="color: rgb(15, 17, 21)"> </span>**<span style="color: rgb(15, 17, 21)">requirements</span>**<span style="color: rgb(15, 17, 21)"> of the software to be built.</span>
>
> #### **<span style="color: rgb(15, 17, 21)">1. 四个主要步骤 / 4 Main Steps</span>**
>
> <span style="color: rgb(15, 17, 21)">需求工程是一个迭代的核心流程，包含四个关键活动。</span>
>
> ##### **<span style="color: rgb(15, 17, 21)">1.1 需求获取 / Elicitation</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">目的</span>**<span style="color: rgb(15, 17, 21)">：理解问题，从利益相关者和其他来源收集需求。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Objective</span>**<span style="color: rgb(15, 17, 21)">: To understand the problem and collect requirements from stakeholders and other sources.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">关键活动</span>**<span style="color: rgb(15, 17, 21)">：</span>
>
>     *   <span style="color: rgb(15, 17, 21)">理解应用领域、识别需求来源、分析利益相关者、选择技术、实际获取需求。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Key Activities</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">Understanding the application domain, identifying sources, analyzing stakeholders, selecting techniques, and eliciting requirements.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">有效技术</span>**<span style="color: rgb(15, 17, 21)">：访谈、协作会议（焦点小组、工作坊）、头脑风暴、场景分析、用户故事。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Effective Techniques</span>**<span style="color: rgb(15, 17, 21)">: Interviews, collaborative sessions (focus groups, workshops), brainstorming, scenario-based analysis, user stories.</span>
>
> ##### **<span style="color: rgb(15, 17, 21)">1.2 分析与协商 / Analysis &#x26; Negotiation</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">目的</span>**<span style="color: rgb(15, 17, 21)">：就问题的</span>**<span style="color: rgb(15, 17, 21)">性质</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">边界</span>**<span style="color: rgb(15, 17, 21)">达成一致，解决歧义与冲突。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Objective</span>**<span style="color: rgb(15, 17, 21)">: To agree upon the </span>**<span style="color: rgb(15, 17, 21)">nature</span>**<span style="color: rgb(15, 17, 21)"> and </span>**<span style="color: rgb(15, 17, 21)">boundaries</span>**<span style="color: rgb(15, 17, 21)"> of the problem, resolving ambiguity and conflicts.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">关键活动</span>**<span style="color: rgb(15, 17, 21)">：</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">概念建模</span>**<span style="color: rgb(15, 17, 21)">：创建明确的领域模型以减少后期意外。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">分析立场</span>**<span style="color: rgb(15, 17, 21)">：分析师可能采取功能主义、社会相对主义、激进结构主义或新人文主义等不同立场来处理需求中的社会与认知维度。</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">确定优先级</span>**<span style="color: rgb(15, 17, 21)">：使用如 MoSCoW（必须有、应该有、可以有、这次不会有）等方法与利益相关者协商需求的优先级。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Key Activities</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">Conceptual Modeling</span>**<span style="color: rgb(15, 17, 21)">: Creating an explicit domain model to reduce later surprises.</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">Analysis Stances</span>**<span style="color: rgb(15, 17, 21)">: The analyst may take different positions (Functionalist, Social-relativist, Radical-structuralist, Neohumanist) to deal with social and cognitive dimensions in requirements.</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">Prioritization</span>**<span style="color: rgb(15, 17, 21)">: Negotiating requirements priorities with stakeholders using methods like MoSCoW (Must have, Should have, Could have, Won‘t have this time).</span>
>
> ##### **<span style="color: rgb(15, 17, 21)">1.3 需求规约 / Specification</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">目的</span>**<span style="color: rgb(15, 17, 21)">：清晰、无歧义地描述问题，形成</span>**<span style="color: rgb(15, 17, 21)">需求规格说明书</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Objective</span>**<span style="color: rgb(15, 17, 21)">: To describe the problem clearly and unambiguously, resulting in a </span>**<span style="color: rgb(15, 17, 21)">requirements specification</span>**<span style="color: rgb(15, 17, 21)">.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">产出</span>**<span style="color: rgb(15, 17, 21)">：需求规格说明书是</span>**<span style="color: rgb(15, 17, 21)">面向客户的合同</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">设计的起点</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Output</span>**<span style="color: rgb(15, 17, 21)">: The requirements specification is a </span>**<span style="color: rgb(15, 17, 21)">contract for the customer</span>**<span style="color: rgb(15, 17, 21)"> and the </span>**<span style="color: rgb(15, 17, 21)">starting point for design</span>**<span style="color: rgb(15, 17, 21)">.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">常见错误</span>**<span style="color: rgb(15, 17, 21)">：需避免</span>**<span style="color: rgb(15, 17, 21)">噪音</span>**<span style="color: rgb(15, 17, 21)">、</span>**<span style="color: rgb(15, 17, 21)">遗漏</span>**<span style="color: rgb(15, 17, 21)">、</span>**<span style="color: rgb(15, 17, 21)">过度规约</span>**<span style="color: rgb(15, 17, 21)">、</span>**<span style="color: rgb(15, 17, 21)">矛盾</span>**<span style="color: rgb(15, 17, 21)">、</span>**<span style="color: rgb(15, 17, 21)">歧义</span>**<span style="color: rgb(15, 17, 21)">、</span>**<span style="color: rgb(15, 17, 21)">前向引用</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">理想化思维</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Common Pitfalls</span>**<span style="color: rgb(15, 17, 21)">: Avoid </span>**<span style="color: rgb(15, 17, 21)">noise, silence, over-specification, contradictions, ambiguity, forward references, and wishful thinking</span>**<span style="color: rgb(15, 17, 21)">.</span>
>
> ##### **<span style="color: rgb(15, 17, 21)">1.4 需求验证 / Validation</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">目的</span>**<span style="color: rgb(15, 17, 21)">：检查需求规格的正确性、完整性、一致性等，确保其准确反映了利益相关者的真实需求。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Objective</span>**<span style="color: rgb(15, 17, 21)">: To inspect the requirement specification w.r.t. correctness, completeness, consistency, etc., ensuring it accurately reflects the real needs of stakeholders.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">方法</span>**<span style="color: rgb(15, 17, 21)">：</span>
>
>     *   <span style="color: rgb(15, 17, 21)">检查来自多个来源的需求以推理其有效性（链接越多越好，但不过度依赖间接链接）。</span>
>     *   <span style="color: rgb(15, 17, 21)">结构化走查、原型评审、制定测试计划、用户验收测试。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Methods</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">Examining requirements from multiple sources to reason about their validity (more links are better, but don‘t rely too much on indirect links).</span>
>     *   <span style="color: rgb(15, 17, 21)">Structured walkthroughs, prototype reviews, creating test plans, user acceptance testing.</span>
>
> #### **<span style="color: rgb(15, 17, 21)">2. 非功能性需求 / Non-Functional Requirements (NFRs)</span>**
>
> <span style="color: rgb(15, 17, 21)">NFRs 至关重要，且在全部需求中常占很高比例。</span>
>
> ##### **<span style="color: rgb(15, 17, 21)">2.1 概述与挑战 / Overview &#x26; Challenges</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">表现多样</span>**<span style="color: rgb(15, 17, 21)">：不同领域、不同类型应用的主要质量属性集不同。一些质量属性可能未被明确定义或模糊不清。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Diverse Manifestations</span>**<span style="color: rgb(15, 17, 21)">: Different domains and application types have different sets of major quality attributes. Some may be undefined or vague.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">核心挑战</span>**<span style="color: rgb(15, 17, 21)">：如果一个质量属性</span>**<span style="color: rgb(15, 17, 21)">无法测量</span>**<span style="color: rgb(15, 17, 21)">或</span>**<span style="color: rgb(15, 17, 21)">无法验证</span>**<span style="color: rgb(15, 17, 21)">，我们就</span>**<span style="color: rgb(15, 17, 21)">无法</span>**<span style="color: rgb(15, 17, 21)">知道软件是否满足它。对于这类属性，只能依赖用户评价。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Core Challenge</span>**<span style="color: rgb(15, 17, 21)">: If a quality attribute </span>**<span style="color: rgb(15, 17, 21)">cannot</span>**<span style="color: rgb(15, 17, 21)"> be measured or verified, we </span>**<span style="color: rgb(15, 17, 21)">cannot</span>**<span style="color: rgb(15, 17, 21)"> know if the software meets it. For such attributes, we can only rely on user evaluations.</span>
>
> ##### **<span style="color: rgb(15, 17, 21)">2.2 类型：质量属性 vs. 约束 / Types: Quality Attributes vs. Constraints</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">质量属性</span>**<span style="color: rgb(15, 17, 21)">：描述软件运行的</span>**<span style="color: rgb(15, 17, 21)">“好坏”</span>**<span style="color: rgb(15, 17, 21)">，通常是可测量的系统属性（如性能、可用性）。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Quality Attributes</span>**<span style="color: rgb(15, 17, 21)">: Describe </span>**<span style="color: rgb(15, 17, 21)">“how well”</span>**<span style="color: rgb(15, 17, 21)"> the software operates, often measurable properties of the system (e.g., performance, availability).</span>
>
> *   **<span style="color: rgb(15, 17, 21)">约束</span>**<span style="color: rgb(15, 17, 21)">：描述项目或系统的</span>**<span style="color: rgb(15, 17, 21)">“限制”</span>**<span style="color: rgb(15, 17, 21)">，是必须遵守的全局性规则（如预算、时间、技术平台）。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Constraints</span>**<span style="color: rgb(15, 17, 21)">: Describe </span>**<span style="color: rgb(15, 17, 21)">“limitations”</span>**<span style="color: rgb(15, 17, 21)"> on the project or system, global rules that must be followed (e.g., budget, schedule, technical platform).</span>
>
> #### **<span style="color: rgb(15, 17, 21)">3. 例子 / Examples</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">非功能性需求（质量属性）示例</span>**<span style="color: rgb(15, 17, 21)">：高可用性（如99.9%正常运行时间）、高性能（如响应时间&#x3C;2秒）、高安全性。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">NFR (Quality Attribute) Example</span>**<span style="color: rgb(15, 17, 21)">: High availability (e.g., 99.9% uptime), high performance (e.g., response time &#x3C; 2s), high security.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">约束示例</span>**<span style="color: rgb(15, 17, 21)">：必须在6个月内完成开发，系统必须运行在Linux平台上，项目预算不超过50万美元。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">Constraint Example</span>**<span style="color: rgb(15, 17, 21)">: Must be developed within 6 months, must run on the Linux platform, project budget must not exceed $500K.</span>

> # **<span style="color: rgb(15, 17, 21)">软件架构：风格、权衡与战术详解（中英对照）</span>**
>
> ## **<span style="color: rgb(15, 17, 21)">一、架构风格：单体架构 vs. 微服务架构 | Architectural Styles: Monolithic vs. Microservices</span>**
>
> ### **<span style="color: rgb(15, 17, 21)">📘 单体架构 | Monolithic Architecture</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，概念类比）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">Architecture is the </span>**<span style="color: rgb(15, 17, 21)">fundamental concepts or properties</span>**<span style="color: rgb(15, 17, 21)"> of a system in its environment embodied in its elements, relationships, and in the principles of its design and evolution.</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">A monolithic architecture can be seen as a </span>**<span style="color: rgb(15, 17, 21)">single-tiered</span>**<span style="color: rgb(15, 17, 21)"> or </span>**<span style="color: rgb(15, 17, 21)">layered</span>**<span style="color: rgb(15, 17, 21)"> style where all functional components are interconnected and bundled into a single unit.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">From the perspective of modularity</span>*<span style="color: rgb(15, 17, 21)">: The system is decomposed into submodules, but they are tightly coupled and deployed as one.</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">*From the perspective of the 4+1 view model*: It often corresponds to a single </span>**<span style="color: rgb(15, 17, 21)">deployment view</span>**<span style="color: rgb(15, 17, 21)"> where the entire application is deployed as a single process.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">架构是系统在其环境中所体现的基本概念或属性，具体化于其元素、关系以及其设计和演化的原则中。</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">单体架构可被视为一种</span>**<span style="color: rgb(15, 17, 21)">单层</span>**<span style="color: rgb(15, 17, 21)">或</span>**<span style="color: rgb(15, 17, 21)">分层</span>**<span style="color: rgb(15, 17, 21)">风格，其中所有功能组件相互连接并捆绑成一个单一单元。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">从模块化的角度：系统被分解为子模块，但它们紧密耦合，并作为一个整体部署。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">从4+1视图模型的角度：它通常对应单一的</span>**<span style="color: rgb(15, 17, 21)">部署视图</span>**<span style="color: rgb(15, 17, 21)">，整个应用程序作为单个进程部署。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 微服务架构 | Microservices Architecture</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，概念类比与推断）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">An architectural style is a description of </span>**<span style="color: rgb(15, 17, 21)">component and connector types</span>**<span style="color: rgb(15, 17, 21)"> and a pattern of their runtime control and/or data transfer.</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">The </span>**<span style="color: rgb(15, 17, 21)">microservices</span>**<span style="color: rgb(15, 17, 21)"> style can be related to the </span>**<span style="color: rgb(15, 17, 21)">microkernel</span>**<span style="color: rgb(15, 17, 21)"> pattern and the </span>**<span style="color: rgb(15, 17, 21)">event-bus/publish-subscribe</span>**<span style="color: rgb(15, 17, 21)"> pattern.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Microkernel pattern</span>*<span style="color: rgb(15, 17, 21)">: A core system (barebone) with plug-in features. Similarly, microservices consist of independent, loosely coupled services that can be developed and deployed separately.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Event-bus pattern</span>*<span style="color: rgb(15, 17, 21)">: Services communicate asynchronously via messages/events, promoting decoupling.</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">In microservices, each service is a </span>**<span style="color: rgb(15, 17, 21)">component</span>**<span style="color: rgb(15, 17, 21)"> with well-defined boundaries, and they interact through lightweight </span>**<span style="color: rgb(15, 17, 21)">connectors</span>**<span style="color: rgb(15, 17, 21)"> (e.g., REST APIs, message queues).</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">架构风格是对组件和连接器类型及其运行时控制和/或数据传输模式的描述。</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">微服务</span>**<span style="color: rgb(15, 17, 21)">风格可以与</span>**<span style="color: rgb(15, 17, 21)">微内核</span>**<span style="color: rgb(15, 17, 21)">模式和</span>**<span style="color: rgb(15, 17, 21)">事件总线/发布-订阅</span>**<span style="color: rgb(15, 17, 21)">模式相关联。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">微内核模式：一个核心系统（骨架）带有插件功能。类似地，微服务由独立的、松散耦合的服务组成，可以单独开发和部署。</span>
>     > *   <span style="color: rgb(15, 17, 21)">事件总线模式：服务通过消息/事件进行异步通信，促进解耦。</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">在微服务中，每个服务都是一个具有明确定义边界的</span>**<span style="color: rgb(15, 17, 21)">组件</span>**<span style="color: rgb(15, 17, 21)">，它们通过轻量级</span>**<span style="color: rgb(15, 17, 21)">连接器</span>**<span style="color: rgb(15, 17, 21)">（如REST API、消息队列）进行交互。</span>
>
> ***
>
> ## **<span style="color: rgb(15, 17, 21)">二、优缺点：可用性与复杂性 | Pros and Cons: Availability &#x26; Complexity</span>**
>
> ### **<span style="color: rgb(15, 17, 21)">📘 单体架构的优缺点 | Pros and Cons of Monolithic Architecture</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，基于质量属性推断）</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">Advantages (Simplicity, Deployability)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Simplicity</span>*<span style="color: rgb(15, 17, 21)">: “A complex problem does not necessarily need a complex solution.” A monolithic design can be simpler to develop and test initially because there is no distributed complexity.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Deployability</span>*<span style="color: rgb(15, 17, 21)">: “Is it easy to deploy the system into its operating environment?” Deploying a single unit is straightforward (manual or automated).</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">Disadvantages (Scalability, Availability, Modifiability)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Scalability</span>*<span style="color: rgb(15, 17, 21)">: “Growth with throughput, users, data volume...” It is harder to scale individual components independently; the entire monolith must be scaled.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Availability</span>*<span style="color: rgb(15, 17, 21)">: A single point of failure. If one component fails, it can bring down the entire system.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Modifiability</span>*<span style="color: rgb(15, 17, 21)">: “Changing one module may lead to changing other N−1 modules.” Tight coupling makes changes riskier and more expensive.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">优点（简洁性，可部署性）</span>**<span style="color: rgb(15, 17, 21)">：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">简洁性：“复杂的问题不一定需要复杂的解决方案。”单体设计在初期开发和测试可能更简单，因为没有分布式复杂性。</span>
>     > *   <span style="color: rgb(15, 17, 21)">可部署性：“将系统部署到其运行环境中是否容易？”部署单个单元是直接的（手动或自动）。</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">缺点（可扩展性，可用性，可修改性）</span>**<span style="color: rgb(15, 17, 21)">：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">可扩展性：“随吞吐量、用户、数据量增长...”难以独立扩展单个组件；必须扩展整个单体。</span>
>     > *   <span style="color: rgb(15, 17, 21)">可用性：存在单点故障。如果一个组件失败，可能导致整个系统崩溃。</span>
>     > *   <span style="color: rgb(15, 17, 21)">可修改性：“更改一个模块可能导致需要更改其他N-1个模块。”紧耦合使得变更风险更大、成本更高。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 微服务架构的优缺点 | Pros and Cons of Microservices Architecture</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，基于质量属性推断）</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">Advantages (Scalability, Availability, Modifiability)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Scalability</span>*<span style="color: rgb(15, 17, 21)">: Each service can be scaled independently based on demand.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Availability</span>*<span style="color: rgb(15, 17, 21)">: “Active redundancy” can be applied per service. Failure of one service does not necessarily bring down the entire system.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Modifiability</span>*<span style="color: rgb(15, 17, 21)">: Services are loosely coupled. “Minimize direct connections between components.” Changes can be made to a service without affecting others.</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">Disadvantages (Complexity, Deployability, Consistency)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Complexity</span>*<span style="color: rgb(15, 17, 21)">: Introduces distributed system complexity: network latency, fault tolerance, data consistency, etc.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Deployability</span>*<span style="color: rgb(15, 17, 21)">: “Coordinate with external dependencies needed?” Deploying and managing many services is more complex (requires orchestration, service discovery).</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Consistency</span>*<span style="color: rgb(15, 17, 21)">: “Is the design conceptually clean and coherent?” Maintaining consistency across services (e.g., data consistency) is challenging.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">优点（可扩展性，可用性，可修改性）</span>**<span style="color: rgb(15, 17, 21)">：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">可扩展性：每个服务可以根据需求独立扩展。</span>
>     > *   <span style="color: rgb(15, 17, 21)">可用性：每个服务可以应用“主动冗余”。一个服务的故障不一定导致整个系统崩溃。</span>
>     > *   <span style="color: rgb(15, 17, 21)">可修改性：服务是松散耦合的。“最小化组件间的直接连接。”可以修改一个服务而不影响其他服务。</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">缺点（复杂性，可部署性，一致性）</span>**<span style="color: rgb(15, 17, 21)">：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">复杂性：引入了分布式系统的复杂性：网络延迟、容错、数据一致性等。</span>
>     > *   <span style="color: rgb(15, 17, 21)">可部署性：“是否需要与外部依赖项协调？”部署和管理许多服务更加复杂（需要编排、服务发现）。</span>
>     > *   <span style="color: rgb(15, 17, 21)">一致性：“设计在概念上是否清晰和连贯？”跨服务维护一致性（例如数据一致性）具有挑战性。</span>
>
> ***
>
> ## **<span style="color: rgb(15, 17, 21)">三、实现高可用的战术 | Tactics for High Availability</span>**
>
> ### **<span style="color: rgb(15, 17, 21)">📘 主动冗余 | Active Redundancy</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT P59, P60）</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">ADD iteratively applies one or more architectural tactics to address a quality attribute issue.</span>**
>     >
>     > <span style="color: rgb(15, 17, 21)">In the context of addressing </span>**<span style="color: rgb(15, 17, 21)">availability</span>**<span style="color: rgb(15, 17, 21)"> within the data tier module:</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Tactic</span>*<span style="color: rgb(15, 17, 21)">: </span>**<span style="color: rgb(15, 17, 21)">Active redundancy</span>**<span style="color: rgb(15, 17, 21)">.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Explanation</span>*<span style="color: rgb(15, 17, 21)">: “address availability => tactic: active redundancy.”</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">Active redundancy involves having multiple instances of a component running simultaneously, all processing requests. If one fails, others can immediately take over.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">ADD迭代地应用一个或多个架构战术来解决质量属性问题。</span>**
>     >
>     > <span style="color: rgb(15, 17, 21)">在解决数据层模块内的</span>**<span style="color: rgb(15, 17, 21)">可用性</span>**<span style="color: rgb(15, 17, 21)">上下文中：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">战术：</span>**<span style="color: rgb(15, 17, 21)">主动冗余</span>**<span style="color: rgb(15, 17, 21)">。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">解释：“解决可用性 => 战术：主动冗余。”</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">主动冗余涉及让一个组件的多个实例同时运行，都处理请求。如果一个失败，其他实例可以立即接管。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 故障检测与恢复 | Fault Detection and Recovery</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，基于“故障相关质量”推断）</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">Reliability</span>**<span style="color: rgb(15, 17, 21)"> and </span>**<span style="color: rgb(15, 17, 21)">Recoverability</span>**<span style="color: rgb(15, 17, 21)"> are key to availability.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">Mean Time Between Failures (MTBF)</span>*<span style="color: rgb(15, 17, 21)"> and </span>*<span style="color: rgb(15, 17, 21)">Mean Time To Recovery (MTTR)</span>*<span style="color: rgb(15, 17, 21)"> are metrics.</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">Tactics include: </span>**<span style="color: rgb(15, 17, 21)">heartbeat monitoring</span>**<span style="color: rgb(15, 17, 21)">, </span>**<span style="color: rgb(15, 17, 21)">health checks</span>**<span style="color: rgb(15, 17, 21)">, and </span>**<span style="color: rgb(15, 17, 21)">automatic failover</span>**<span style="color: rgb(15, 17, 21)">.</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">From the design perspective, we can incorporate </span>**<span style="color: rgb(15, 17, 21)">state replication</span>**<span style="color: rgb(15, 17, 21)"> and </span>**<span style="color: rgb(15, 17, 21)">transaction logging</span>**<span style="color: rgb(15, 17, 21)"> to facilitate recovery.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">可靠性</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">可恢复性</span>**<span style="color: rgb(15, 17, 21)">是可用性的关键。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">平均无故障时间（MTBF）和平均恢复时间（MTTR）是度量指标。</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">战术包括：</span>**<span style="color: rgb(15, 17, 21)">心跳监控</span>**<span style="color: rgb(15, 17, 21)">、</span>**<span style="color: rgb(15, 17, 21)">健康检查</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">自动故障转移</span>**<span style="color: rgb(15, 17, 21)">。</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">从设计角度，我们可以采用</span>**<span style="color: rgb(15, 17, 21)">状态复制</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">事务日志</span>**<span style="color: rgb(15, 17, 21)">来促进恢复。</span>
>
> ***
>
> ## **<span style="color: rgb(15, 17, 21)">四、冗余设计模式 | Redundancy Design Patterns</span>**
>
> ### **<span style="color: rgb(15, 17, 21)">📘 企业集成模式中的冗余 | Redundancy in Enterprise Integration Patterns</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT P68-P70）</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">Scatter-Gather</span>**<span style="color: rgb(15, 17, 21)"> pattern (a form of redundancy for reliability and performance):</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">“Send a message to a publish-subscribe channel. Interested recipients subscribe to a ‘topic’. Aggregator collects individual response messages.”</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">This pattern uses multiple providers (redundancy) to process a request and aggregates the results (e.g., best quote). It improves </span>**<span style="color: rgb(15, 17, 21)">availability</span>**<span style="color: rgb(15, 17, 21)"> and </span>**<span style="color: rgb(15, 17, 21)">performance</span>**<span style="color: rgb(15, 17, 21)">.</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">Aggregator</span>**<span style="color: rgb(15, 17, 21)"> pattern:</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">“Collects and stores messages until a complete set has been received (completeness condition). Publishes a single message created from the individual messages (aggregation algorithm).”</span>
>     > *   <span style="color: rgb(15, 17, 21)">Used in conjunction with redundancy to consolidate responses.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> **<span style="color: rgb(15, 17, 21)">分散-收集</span>**<span style="color: rgb(15, 17, 21)">模式（一种用于可靠性和性能的冗余形式）：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">“将消息发送到发布-订阅通道。感兴趣的接收者订阅一个‘主题’。聚合器收集各个响应消息。”</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">该模式使用多个提供者（冗余）来处理请求并聚合结果（例如最佳报价）。它提高了</span>**<span style="color: rgb(15, 17, 21)">可用性</span>**<span style="color: rgb(15, 17, 21)">和</span>**<span style="color: rgb(15, 17, 21)">性能</span>**<span style="color: rgb(15, 17, 21)">。</span>
>     >
>     > **<span style="color: rgb(15, 17, 21)">聚合器</span>**<span style="color: rgb(15, 17, 21)">模式：</span>
>     >
>     > *   <span style="color: rgb(15, 17, 21)">“收集并存储消息，直到收到完整的集合（完成条件）。发布一条由单个消息创建而成的单一消息（聚合算法）。”</span>
>     > *   <span style="color: rgb(15, 17, 21)">与冗余结合使用以整合响应。</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 冗余作为战术的模式化实现 | Patterned Implementation of Redundancy as a Tactic</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT，综合）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">Redundancy can be implemented using patterns such as:</span>
>     >
>     > *   **<span style="color: rgb(15, 17, 21)">Leader-Follower (Master-Slave)</span>**<span style="color: rgb(15, 17, 21)">: “Master-slave: Master dispatches tasks to multiple slaves and collects acknowledgments.” If the master fails, a follower can be promoted.</span>
>     >
>     > *   **<span style="color: rgb(15, 17, 21)">Active-Active</span>**<span style="color: rgb(15, 17, 21)">: Multiple identical components serve requests simultaneously, sharing the load and providing failover.</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">These patterns address </span>**<span style="color: rgb(15, 17, 21)">availability</span>**<span style="color: rgb(15, 17, 21)"> by ensuring that there is no single point of failure.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">冗余可以通过以下模式实现：</span>
>     >
>     > *   **<span style="color: rgb(15, 17, 21)">领导者-跟随者（主-从）</span>**<span style="color: rgb(15, 17, 21)">：“主-从：主节点向多个从节点分派任务并收集确认。”如果主节点失败，可以提升一个从节点。</span>
>     >
>     > *   **<span style="color: rgb(15, 17, 21)">主动-主动</span>**<span style="color: rgb(15, 17, 21)">：多个相同的组件同时服务请求，共享负载并提供故障转移。</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">这些模式通过确保没有单点故障来解决</span>**<span style="color: rgb(15, 17, 21)">可用性</span>**<span style="color: rgb(15, 17, 21)">问题。</span>

### Technical Debt

*   Definition: "Quick & dirty" vs. Strategic
*   Dimensions: Intentional vs. Unintentional
*   Short-term vs. Long-term impacts
*   Management: Backlog vs. Code Comments

> # **<span style="color: rgb(15, 17, 21)">技术债务详解（中英对照）</span>**
>
> ## **<span style="color: rgb(15, 17, 21)">一、技术债务基础 | Technical Debt Fundamentals</span>**
>
> ### **<span style="color: rgb(15, 17, 21)">📘 定义 | Definition</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">Technical debt (TD) is a metaphor originally proposed in 1992. The metaphor is that doing things in a “quick and dirty” way creates a technical debt (TD).</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">It is a design or construction approach that is expedient in the short term but that creates a technical context in which </span>**<span style="color: rgb(15, 17, 21)">the same work will cost more to do later than it would cost to do now</span>**<span style="color: rgb(15, 17, 21)"> (including increased cost over time).</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">技术债务是一个隐喻，最初于1992年提出。该隐喻是指：用“快速而肮脏”的方式做事会产生技术债务。</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">它是一种在短期内容易实现的设计或构建方法，但这种方法会创造一个技术背景，使得</span>**<span style="color: rgb(15, 17, 21)">以后完成相同工作的成本将高于现在的成本</span>**<span style="color: rgb(15, 17, 21)">（包括随时间增加的成本）。</span>
>
> #### **<span style="color: rgb(15, 17, 21)">📘 “利息支付”示例 | Examples of "Interest Payments"</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT）</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">E.g., legacy code lacking unit tests leads to longer time to develop, test, and debug.</span>
>     > *   <span style="color: rgb(15, 17, 21)">E.g., simplistic design not adapting well to changes in the environment means an environment change, which seemed trivial, demands a large amount of code rewriting.</span>
>     > *   <span style="color: rgb(15, 17, 21)">E.g., bug-ridden system leads to high product support costs (a non-software form of interest payment).</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> *   <span style="color: rgb(15, 17, 21)">例如，遗留代码缺乏单元测试，导致新开发、测试和调试需要更长时间。</span>
>     > *   <span style="color: rgb(15, 17, 21)">例如，过于简单的设计不能很好地适应环境变化，看似简单的环境变更需要大量的代码重写。</span>
>     > *   <span style="color: rgb(15, 17, 21)">例如，漏洞百出的系统导致高昂的产品支持成本（非软件形式的利息支付）。</span>
>
> #### **<span style="color: rgb(15, 17, 21)">📘 技术债务的反例 | What is NOT Technical Debt?</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">Many kinds of delayed or incomplete work are not technical debt: e.g., feature backlogs, deferred features, cut features.</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">In general, if </span>**<span style="color: rgb(15, 17, 21)">“the same (development) work will not cost more to do later than it would cost to do now”</span>**<span style="color: rgb(15, 17, 21)">, then it is not a technical debt.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">许多类型的延迟或未完成的工作并不属于技术债务：例如，功能待办列表、延迟的功能、被削减的功能。</span>
>     >
>     > <span style="color: rgb(15, 17, 21)">一般来说，如果</span>**<span style="color: rgb(15, 17, 21)">“相同的（开发）工作在以后完成并不会比现在完成成本更高”</span>**<span style="color: rgb(15, 17, 21)">，那么它就不是技术债务。</span>
>
> ***
>
> ## **<span style="color: rgb(15, 17, 21)">二、技术债务的维度：有意与无意 | Dimensions of TD: Intentional vs. Unintentional</span>**
>
> ### **<span style="color: rgb(15, 17, 21)">📘 有意技术债务 | Intentional Technical Debt</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">Debt incurred by intention (we intend to commit to do it).</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">E.g.</span>*<span style="color: rgb(15, 17, 21)">: “If we can’t make this release, there won’t be a second one. Let’s focus on coding, doing the minimal testing. No documentation. Run the system with the users in the first few days after the release, rather than train them ahead of time.”</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">E.g.</span>*<span style="color: rgb(15, 17, 21)">: “We don’t have time to build generic support for multiple platforms. We support iOS now and will add support for Android, etc., later. Code only needs to work fine for iOS apps.”</span>
>
> *   **<span style="color: rgb(15, 17, 21)">中文翻译</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">故意产生的债务（我们有意决定这么做）。</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">例如</span>*<span style="color: rgb(15, 17, 21)">：“如果我们无法完成这个版本，就不会有第二个版本了。让我们专注于编码，只做最少的测试。不写文档。在版本发布后的头几天与用户一起操作系统，而不是提前培训他们。”</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">例如</span>*<span style="color: rgb(15, 17, 21)">：“我们没有时间构建对多平台的通用支持。我们现在先支持 iOS，以后再增加对 Android 等的支持。代码只需要在 iOS 应用上运行良好即可。”</span>
>
> ### **<span style="color: rgb(15, 17, 21)">📘 无意技术债务 | Unintentional Technical Debt</span>**
>
> *   **<span style="color: rgb(15, 17, 21)">英文原文（源自PPT）</span>**<span style="color: rgb(15, 17, 21)">：</span>> <span style="color: rgb(15, 17, 21)">Debt incurred unintentionally due to low-quality work.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">E.g.</span>*<span style="color: rgb(15, 17, 21)">: A junior programmer writes poor code.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">E.g.</span>*<span style="color: rgb(15, 17, 21)">: A new member/contractor writes code that does not conform to coding standards.</span>
>     >
>     > *   *<span style="color: rgb(15, 17, 21)">E.g.</span>*<span style="color: rgb(15, 17, 21)">: A major design strategy (e.g., deciding to use a model-view-controller architecture to connect all major modules in the software) turned out poorly.</span>
>

>     > *   *<span style="color: rgb(15, 17, 21)">例如</span>*<span style="color: rgb(15, 17, 21)">：一个重大的设计策略（例如，决定使用模型-视图-控制器架构来连接软件中的所有主要模块）结果效果很差。</span>

### Fundamental Principles

*   Abstraction

> ### **<span style="color: rgb(15, 17, 21)">基本设计原则 / Fundamental Design Principles</span>**
>
> #### **<span style="color: rgb(15, 17, 21)">1. 抽象 / Abstraction</span>**
>
> <span style="color: rgb(15, 17, 21)">抽象是编程理论中的一个基本概念，旨在避免信息或人力的重复。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">定义 / Definition</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">抽象旨在</span>**<span style="color: rgb(15, 17, 21)">将接口与实现分离 (separate interface from implementation)</span>**<span style="color: rgb(15, 17, 21)">，并具有一个明确的</span>**<span style="color: rgb(15, 17, 21)">目的 (purpose)</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
>     *   <span style="color: rgb(15, 17, 21)">Abstraction aims to </span>**<span style="color: rgb(15, 17, 21)">separate interface from implementation with a purpose</span>**<span style="color: rgb(15, 17, 21)">.</span>
>
> *   **<span style="color: rgb(15, 17, 21)">功能性抽象 (Functional Purpose)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">将一段具有</span>**<span style="color: rgb(15, 17, 21)">明确定义的目的 (well-defined purpose)</span>**<span style="color: rgb(15, 17, 21)"> 的代码 </span>`C`（例如，对整数数组进行排序的代码）放入一个具有函数签名 `F`（例如，`int[] sort(int[] balls)`）的函数体中。我们称 `F` 是 `C` 的抽象 (`F` is an abstraction of `C`)。
>
>     *   <span style="color: rgb(15, 17, 21)">To abstract the </span>**<span style="color: rgb(15, 17, 21)">well-defined functional purpose</span>**<span style="color: rgb(15, 17, 21)"> of a piece of code </span>`C` by putting it into the body of a function with signature `F`. We say that `F` is an abstraction of `C`.
>
>     *   **<span style="color: rgb(15, 17, 21)">好处 / Benefits</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>         *   <span style="color: rgb(15, 17, 21)">客户端代码无需理解 </span>`C` 的细节，只需依赖抽象 `F` 及其明确定义的功能目的。
>
>         *   <span style="color: rgb(15, 17, 21)">对 </span>`C` 的更改可以局限在 `F` 内部，对客户端代码应是透明的。
>
>         *   <span style="color: rgb(15, 17, 21)">客户端代码</span>**<span style="color: rgb(15, 17, 21)">更具可读性 (more readable)</span>**<span style="color: rgb(15, 17, 21)">，因其逻辑可专注于实现自身目标。</span>
>
>         *   <span style="color: rgb(15, 17, 21)">Client code </span>`U` becomes **more readable** as its code logic can focus on achieving its own purpose.
>
> *   **<span style="color: rgb(15, 17, 21)">泛化 (Generalization)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">抽象还包括 </span>**<span style="color: rgb(15, 17, 21)">(1) 寻找相似性或共同点，以及 (2) 忽略不重要的差异</span>**<span style="color: rgb(15, 17, 21)">的过程。</span>
>
>     *   <span style="color: rgb(15, 17, 21)">Abstraction can include the process of </span>**<span style="color: rgb(15, 17, 21)">(1) finding similarities and/or common aspects and (2) forgetting unimportant differences</span>**<span style="color: rgb(15, 17, 21)">.</span>
>
>     *   <span style="color: rgb(15, 17, 21)">对于一组具有</span>*<span style="color: rgb(15, 17, 21)">相似</span>*<span style="color: rgb(15, 17, 21)">功能目的的代码 </span>`{C1, ..., Cn}`，可以创建一个共同的抽象 `F`。`F` 的功能目的是各个 `C` 的功能目的的**共同 (common)** 部分。
>
> *   **<span style="color: rgb(15, 17, 21)">数据抽象 (Data Abstraction)</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">将</span>**<span style="color: rgb(15, 17, 21)">与数据结构关键相关 (critically related)</span>**<span style="color: rgb(15, 17, 21)"> 的函数抽象并分组到一个集合中，该集合代表了数据结构的</span>**<span style="color: rgb(15, 17, 21)">单一功能目的 (single functional purpose)</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
>     *   <span style="color: rgb(15, 17, 21)">To model the operators on a data structure as a set of </span>*<span style="color: rgb(15, 17, 21)">related</span>*<span style="color: rgb(15, 17, 21)"> functions (called </span>**<span style="color: rgb(15, 17, 21)">interfaces</span>**<span style="color: rgb(15, 17, 21)">), while concealing the data structure implementations.</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">好的例子 / Good Example</span>**<span style="color: rgb(15, 17, 21)">: 栈 (Stack) 的操作 (</span>`push`, `pop`, `isEmpty` 等) 完整地建模了后进先出 (LIFO) 的行为，并隐藏了内部实现（如使用数组还是链表）。
>
>     *   **<span style="color: rgb(15, 17, 21)">坏的例子 / Bad Example</span>**<span style="color: rgb(15, 17, 21)">: 一个名为“栈”的接口提供了类似数组的随机访问方法（如 </span>`get(int i)`, `set(int x, int i)`），这暴露了内部实现，破坏了栈数据结构的完整性。
>
> #### **<span style="color: rgb(15, 17, 21)">2. 信息隐藏 / Information Hiding</span>**
>
> <span style="color: rgb(15, 17, 21)">信息隐藏旨在从数据结构的接口中移除不必要的细节。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">定义 / Definition</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">信息隐藏是</span>**<span style="color: rgb(15, 17, 21)">从数据结构的接口中移除细节 (remove details from the interfaces of a data structure)</span>**<span style="color: rgb(15, 17, 21)">（通过互相关联这些接口来实现）。</span>
>
>     *   <span style="color: rgb(15, 17, 21)">Information hiding is to </span>**<span style="color: rgb(15, 17, 21)">remove details from the interfaces of a data structure</span>**<span style="color: rgb(15, 17, 21)"> (by interrelating these interfaces).</span>
>
>     *   <span style="color: rgb(15, 17, 21)">接口的选择应</span>**<span style="color: rgb(15, 17, 21)">尽可能少地暴露其内部工作细节 (reveal as little as possible about its internal workings)</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">关键思想 / Key Idea</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">应评估接口中操作参数所指定的细节层次是否</span>**<span style="color: rgb(15, 17, 21)">必要 (essential)</span>**<span style="color: rgb(15, 17, 21)">。更抽象的接口通常更好。</span>
>
>     *   <span style="color: rgb(15, 17, 21)">We should ask whether the level of details specified in the parameters of an operator in an interface is essential.</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">例子 / Example</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>         *   `boolean canVote1(int HKID, int age, int district_code)` 暴露了选民的所有详细信息。
>         *   `boolean canVote2(Voter v)` 仅接受一个 `Voter` 对象，隐藏了具体的判断标准。`canVote1()` 比 `canVote2()` 暴露了更多细节 (`canVote1()` reveals more details than `canVote2()`)。
>
> #### **<span style="color: rgb(15, 17, 21)">3. 封装 / Encapsulation</span>**
>
> <span style="color: rgb(15, 17, 21)">封装旨在限制客户端代码获取实现细节的知识。</span>
>
> *   **<span style="color: rgb(15, 17, 21)">定义 / Definition</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">封装是指</span>**<span style="color: rgb(15, 17, 21)">限制（客户端代码）获取有关代码实现细节的知识 (restrict gaining knowledge about the implementation details of the code to its client code)</span>**<span style="color: rgb(15, 17, 21)">。</span>
>
>     *   Encapsulation refers to ensuring outsiders cannot gain knowledge about certain information internal t<span style="color: rgb(15, 17, 21)">o </span>`X`.
>
>     *   `X` 可以是一个类、模块、文件、数据记录等，也代表内部数据、算法等。
>
> *   **<span style="color: rgb(15, 17, 21)">实现方式 / How to Achieve</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">如果我们能用最少的“已知工作方式”来描述数据结构的操作的功能目的，并且功能签名没有规定超出该功能目的的内容，我们就实现了良好的封装。</span>
>
>     *   <span style="color: rgb(15, 17, 21)">We achieve good encapsulation if we can describe the functional purpose using the least “working known-how” of the operators of a data structure and the functional signature does not specify more than what the functional purpose states.</span>
>
>     *   **<span style="color: rgb(15, 17, 21)">访问控制 (Access Control)</span>**<span style="color: rgb(15, 17, 21)">: 通过控制类中每个数据成员或方法的可见性（如使用 </span>`private`, `public`）来**提高 (increase)** 封装级别。
>
>     *   **<span style="color: rgb(15, 17, 21)">输入/输出参数 (Input/Output Parameters)</span>**<span style="color: rgb(15, 17, 21)">: 同样应控制输入/输出参数，避免暴露内部状态。例如，提供 </span>`getSize()` 和 `setSize()` 方法可能会让客户端获知并篡改内部 `size` 变量，破坏数据完整性。
>
> *   **<span style="color: rgb(15, 17, 21)">违反封装的例子 / Example of Encapsulation Violation</span>**<span style="color: rgb(15, 17, 21)">:</span>
>
>     *   <span style="color: rgb(15, 17, 21)">如果栈 (</span>`Stack`) 的内部实现依赖于一个数组 (`MyArray`)，并且将这个内部数组的引用暴露给了客户端代码，客户端就可以直接操作该数组，从而**破坏 (corrupt)** 栈的内部状态和完整性。
>
>     *   <span style="color: rgb(15, 17, 21)">The client code can change the internal data structure for the stack, which </span>**<span style="color: rgb(15, 17, 21)">corrupts</span>**<span style="color: rgb(15, 17, 21)"> the stack's integrity.</span>

### SOLID Principles

> ## **SOLID 原则详解**
>
> ### **S - 单一职责原则 | Single-Responsibility Principle (SRP)**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（PPT P33）**：> A responsibility as *a logical reason* to change the code. The principle states that a class should have one and only one reason to be rewritten. A responsibility is not the assigned functionality of the class.
>
> *   **中文翻译**：> 职责是修改代码的一个**逻辑原因**。该原则表明，一个类应该有且只有一个被重写的理由。职责不是该类所分配的功能。
>
> #### **📘 解释与示例 | Explanation & Example**
>
> *   **英文原文（PPT P33）**：> E.g., a class `SortedList` takes up the “logical” responsibility of “modeling a sorted sequence.” Anything related to changing how a sorted sequence is modeled, it should be done within `SortedList`.\
>     > If SRP is respected, then the responsibility means delivering the functional purpose of the class.
>
> *   **中文翻译**：> 例如，`SortedList` 类承担了“建模有序序列”的“逻辑”职责。任何与修改有序序列建模方式相关的事，都应在 `SortedList` 内完成。\
>     > 如果遵循 SRP，那么职责就意味着实现该类的功能目的。
>
> #### **📘 为何要改变类代码？ | Why Do We Want to Change the Code in the Class?**
>
> *   **英文原文（PPT P34）**：> Why do we want to change the code in the class Stack?
>     >
>     > *   Implement a faster algorithm for the same purpose?
>     > *   Fix bugs in the existing code?
>     > *   Change to use another data structure/framework instead of int\[]?
>     > *   Implement missing requirements required to model Stack?
>
> *   **中文翻译**：> 我们为什么想要修改类 Stack 的代码？
>     >
>     > *   为实现相同目的而采用更快的算法？
>     > *   修复现有代码中的错误？
>     > *   更改为使用另一种数据结构/框架而不是 int\[]？
>     > *   实现建模 Stack 所需的缺失需求？
>
> ***
>
> ### **O - 开闭原则 | Open-Closed Principle (OCP)**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（PPT P35）**：> Open-closed principle requires a code pattern to meet two conditions:
>     >
>     > *   Open for extension, **and**
>     > *   Close for modification\
>     >     The basic rule of OCP is:\
>     >     (1) extend without modifying the existing code or\
>     >     (2) refine the implementation without changing the functional purpose and input/output arguments.
>
> *   **中文翻译**：> 开闭原则要求代码模式满足两个条件：
>     >
>     > *   对扩展开放，**且**
>     > *   对修改封闭\
>     >     OCP 的基本规则是：\
>     >     (1) 不修改现有代码即可扩展，或\
>     >     (2) 在不改变功能目的和输入/输出参数的情况下改进实现。
>
> #### **📘 反例与正例 | Counter-Example & Example**
>
> *   **反例（PPT P36）**：> 使用 `switch` 语句和 `enum` 类型来判断员工类型并执行不同的支付逻辑。要添加新类型，必须修改 `enum` 和 `switch` 语句。
>
> *   **正例（PPT P37）**：> 使用多态，定义抽象的 `SalaryType` 类及其子类（`MonthlySalaryType`, `DailySalaryType`）。`Employee` 类的 `receiveSalary()` 方法调用 `type.doThis()`。要添加新类型，只需创建新的 `SalaryType` 子类，无需修改 `Employee` 类。
>
> *   **另一个反例（PPT P38）**：> 即使使用了多态，但在 `receiveSalary()` 方法中使用 `instanceof` 进行类型判断，也违反了 OCP。添加新子类仍需修改该方法。
>
> ***
>
> ### **L - 里氏替换原则 | Liskov Substitution Principle (LSP)**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（PPT P41, P42）**：> Suppose `S` is a subclass of `T`. `S` can be reasoned in every client code `C` in the same way as `T` is reasoned in `C`. In this case, LSP is followed by the pair of `S` and `T`.\
>     > E.g., `S` may refine or extend `T`.
>     >
>     > *   `S` refines `T`. E.g., `S` and `T` compute the same result, but `S` implements a better algorithm or uses a better data structure.
>     > *   `S` extends `T`. E.g., `S` inherits everything from `T` and does extra things.
>
> *   **中文翻译**：> 假设 `S` 是 `T` 的子类。在任何客户端代码 `C` 中，`S` 应该可以以与 `T` 相同的方式被推理和使用。此时，`S` 和 `T` 遵循了 LSP。\
>     > 例如，`S` 可以优化或扩展 `T`。
>     >
>     > *   `S` 优化 `T`。例如，`S` 和 `T` 计算结果相同，但 `S` 使用更好的算法或数据结构。
>     > *   `S` 扩展 `T`。例如，`S` 继承 `T` 的所有内容并做额外的事情。
>
> #### **📘 示例与反例 | Example & Counter-Example**
>
> *   **示例（PPT P43）**：> `Manager` 是 `Employee` 的子类，不仅接收薪水，还管理下属。这符合 LSP，因为 `Manager` 可以完全替代 `Employee` 的行为（接收薪水）并进行了扩展。
>
> *   **反例（PPT P44）**：> `Employee` 是 `Manager` 的子类。因为普通员工不管理他人，所以 `manageEmployee()` 方法被一个空方法（dummy method）覆盖。这违反了 LSP，因为 `Employee` 无法真正替代 `Manager` 的管理职责。
>
> *   **概念辨析（PPT P41, P42）**：> *   鸭子是鸟的符合 LSP 的子类。
>     > *   整数**不是**浮点数的子类型（因为不具备分数运算的特性）。
>
> ***
>
> ### **I - 接口隔离原则 | Interface Segregation Principle (ISP)**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（PPT P45）**：> ISP states that a client (of an interface) should not be forced to depend on methods (i.e., implement the methods) it does not use.\
>     > Warning: ISP has no theoretical basis. It is a handy concept to avoid some problematic designs.
>
> *   **中文翻译**：> ISP 规定，客户端不应被迫依赖它不使用的方法（即实现这些方法）。\
>     > 注意：ISP 没有理论基础。它是一个实用概念，用于避免某些有问题的设计。
>
> #### **📘 反例与正例 | Counter-Example & Example**
>
> *   **反例（PPT P46）**：> `ColorScheme` 接口包含 `renderInRGB`, `renderInYUV`, `renderInHSV` 三个方法。`myDisplay` 类只需要 RGB 渲染，但被迫实现另外两个空方法。
>
> *   **正例（PPT P47）**：> 将 `ColorScheme` 拆分为三个独立的接口：`RGBColorScheme`, `YUVColorScheme`, `HSVColorScheme`。`myDisplay` 类只需实现 `RGBColorScheme` 接口。
>
> *   **另一个反例（PPT P48）**：> 一个通用接口 `X` 包含 `f`, `g`, `h` 方法。不同子类（A, B, C）对这些方法的调用顺序有隐含依赖（如 B 应先调f后调g，C 应先调g后调f），导致客户端难以正确使用，且违反了 ISP 的精神。
>
> ***
>
> ### **D - 依赖倒置原则 | Dependency Inversion Principle (DIP)**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（PPT P49）**：> DIP requires both conditions below to be held:
>     >
>     > 1.  High-level modules should not depend on low-level modules. Both should depend on abstractions.
>     > 2.  Abstractions should not depend on details. Details should depend on abstractions.
>
> *   **中文翻译**：> DIP 要求满足以下两个条件：
>     >
>     > 1.  高层模块不应依赖低层模块。两者都应依赖抽象。
>     > 2.  抽象不应依赖细节。细节应依赖抽象。
>
> #### **📘 依赖注入 | Dependency Injection**
>
> *   **英文原文（PPT P50）**：> At the top level, PolicyService is an interface, and yet a Policy object needs to call a concrete object. We may inject a dependency Mechanism object to the Policy object.\
>     > Pass an object when constructing a Policy object. We need a Mechanism factory to create the former object.
>
> *   **中文翻译**：> 在顶层，PolicyService 是一个接口，但 Policy 对象需要调用一个具体对象。我们可以将一个依赖的 Mechanism 对象注入到 Policy 对象中。\
>     > 在构造 Policy 对象时传入一个对象。我们需要一个 Mechanism 工厂来创建前者对象。

> ## **代码异味详解**
>
> ### **📘 定义与影响 | Definition and Impact**
>
> *   **英文原文（PPT P52, P53）**：> **Code smell** is a violation of a design principle and causes an adverse effect to the code.\
>     > Code smell is a violation of a design principle and (potentially) harms the code.\
>     > Novice developers tend to find it more difficult to judge the presence of them in their code.

>     > 新手开发者往往更难判断其代码中是否存在这些异味。
>
> *   **研究结果（PPT P57）**：> A study shows that classes with code smells change more often and incur more bug-fixing patches. Classes with more code smells change more often than classes with less code smells. It means that the code degraded.
>

> | **复杂类**       | Complex Class (CC)                   | 类具有高复杂度。                     |

> ### **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">1. Magic Numbers（魔数）</span></span>**
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：在代码中直接使用没有解释意义的数字字面量，使得代码难以理解和维护。这些数字通常代表某种业务逻辑或配置值，但没有明确的命名。</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">英文解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：Magic Numbers are numeric literals used directly in code without explanation, making the code hard to understand and maintain. They often represent business logic or configuration values but lack meaningful names.</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">示例</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：</span></span>
>
> ```
> // 坏例子：魔数
> public double calculateArea(double radius) {
>     return 3.14159 * radius * radius; // 3.14159 是魔数
> }
>
> // 好例子：使用常量
> public static final double PI = 3.14159;
> public double calculateArea(double radius) {
>     return PI  radius  radius;
> }
> ```
>

> ### **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">2. Hardcoding（写死）</span></span>**
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：将可能会变化的值（如文件路径、URL、数据库连接字符串、业务规则阈值等）直接写入代码中，而不是通过配置、常量或参数传递。这会导致代码缺乏灵活性，修改时需改动源代码。</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">英文解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：Hardcoding refers to embedding values that may change (like file paths, URLs, database connection strings, or business rule thresholds) directly into the source code, rather than using configuration files, constants, or parameters. This reduces flexibility and requires code changes when values need to be updated.</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">示例</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：</span></span>
>
> ```
> // 坏例子：硬编码
> public void loadConfig() {
>     String filePath = "C:/project/config.properties"; // 硬编码路径
>     // 读取文件...
> }
>
> // 好例子：从配置或常量读取
> public static final String CONFIG_PATH = "config.properties"; // 或从环境变量读取
> public void loadConfig() {
>     String filePath = System.getProperty("config.path", CONFIG_PATH);
>     // 读取文件...
> }
> ```
>

> ### **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">3. Long Methods（过长方法）</span></span>**
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：一个方法或函数的代码行数过多，通常意味着它承担了过多的职责，违反了“单一职责原则”。这样的方法难以理解、测试和维护。</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">英文解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：Long Methods are functions or methods that contain too many lines of code, often indicating they have too many responsibilities and violate the Single Responsibility Principle. Such methods are hard to understand, test, and maintain.</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">示例</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：</span></span>
>
> ```
> // 坏例子：过长方法
> public void processOrder(Order order) {
>     // 验证订单（20行）
>     // 计算价格（15行）
>     // 更新库存（25行）
>     // 发送通知（10行）
>     // 记录日志（5行）
>     // 总计约75行
> }
>
> // 好例子：拆分为多个小方法
> public void processOrder(Order order) {
>     validateOrder(order);
>     calculatePrice(order);
>     updateInventory(order);
>     sendNotification(order);
>     logOrder(order);
> }
> private void validateOrder(Order order) { /* ... */ }
> private void calculatePrice(Order order) { /* ... */ }
> // ... 其他小方法
> ```
>

> ### **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">4. Poor Naming（命名不当）</span></span>**
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：使用不清晰、不准确或误导性的名称来命名变量、函数、类或方法。这会导致代码可读性差，其他开发者难以理解其意图。</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">英文解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：Poor Naming refers to using unclear, inaccurate, or misleading names for variables, functions, classes, or methods. This reduces code readability and makes it difficult for other developers to understand the intent.</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">示例</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：</span></span>
>
> ```
> // 坏例子：命名不当
> int d; // 表示天数？距离？直径？
>     void p() { // 表示打印？处理？解析？
>     // ...
> }
> List a; // 什么列表？
>
> // 好例子：清晰命名
> int daysUntilExpiry;
> void printReport() {
>     // ...
> }
> List<Customer> activeCustomers;
> ```
>

> ### **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">5. Code Duplication（代码重复）</span></span>**
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：相同的或非常相似的代码结构出现在代码库的多个地方。这会增加维护成本，因为任何逻辑修改都需要在多个地方进行，容易遗漏并引入错误。</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">英文解释</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：Code Duplication occurs when the same or very similar code structures appear in multiple places in the codebase. This increases maintenance costs because any logic change must be made in multiple locations, making it easy to miss updates and introduce errors.</span></span>
>
> **<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">示例</span></span>**<span style="color: rgb(15, 17, 21)"><span style="background-color: rgb(255, 255, 255)">：</span></span>
>
> ```
> // 坏例子：重复的验证逻辑
> public void registerUser(User user) {
>     if (user.getEmail() == null || user.getEmail().isEmpty()) {
>         throw new IllegalArgumentException("Email is required");
>     }
>     // 注册逻辑...
> }
> public void updateProfile(User user) {
>     if (user.getEmail() == null || user.getEmail().isEmpty()) {
>         throw new IllegalArgumentException("Email is required");
>     }
>     // 更新逻辑...
> }
>
> // 好例子：提取公共方法
> public void validateEmail(User user) {
>     if (user.getEmail() == null || user.getEmail().isEmpty()) {
>         throw new IllegalArgumentException("Email is required");
>     }
> }
> public void registerUser(User user) {
>     validateEmail(user);
>     // 注册逻辑...
> }
> public void updateProfile(User user) {
>     validateEmail(user);
>     // 更新逻辑...
> }
> ```

> ### **📘 Monkey (Android)**
>
> #### **📘 定义与概述 | Definition & Overview**
>
> *   **英文原文（源自PPT P27及上下文）**：> The Monkey tool emulates user interaction with an app. It generates and injects gestures commands or system events based on a predefined event frequency distribution into the app’s event input stream. The tool watches for exceptional conditions, e.g., the app crashing or throwing an exception.> Monkey is a fuzzing tool for Android GUI testing. Developers can specify the distribution of kind of events to define a test profile.
>
> *   **中文翻译**：> Monkey 工具模拟用户与应用程序的交互。它根据预定义的事件频率分布生成手势命令或系统事件，并将其注入到应用程序的事件输入流中。该工具监视异常情况，例如应用程序崩溃或抛出异常。> Monkey 是用于 Android GUI 测试的模糊测试工具。开发者可以指定不同类型事件的分布来定义测试配置文件。
>
> #### **📘 关键特性与用法 | Key Characteristics & Usage**
>
> *   **英文原文（源自PPT P27, P28）**：> *   **Default Events**: Monkey has a set of default events (e.g., TOUCH, MOTION, TRACKBALL, NAV) with predefined percentages.
>     > *   **Throttle**: The tool can vary the “throttle”, i.e., the time delays between events; by default, there is no delay between events (throttle = 0).
>     > *   **Command Line**: Started via ADB (Android Debug Bridge) with commands like `adb shell monkey -p your.package.name -v 500`.
>     > *   **Effectiveness**: Good at finding crashes quickly, but manual testing often achieves better code coverage.
>
> *   **中文翻译**：> *   **默认事件**：Monkey 有一套默认事件（如 TOUCH, MOTION, TRACKBALL, NAV）及其预定义的百分比。
>     > *   **延迟**：该工具可以改变“节流阀”，即事件之间的时间延迟；默认情况下，事件之间没有延迟（throttle = 0）。
>     > *   **命令行**：通过 ADB (Android Debug Bridge) 启动，命令如 `adb shell monkey -p your.package.name -v 500`。
>     > *   **有效性**：擅长快速发现崩溃，但手动测试通常能实现更好的代码覆盖率。
>
> ### **📘 American Fuzzy Lop (AFL)**
>
> #### **📘 定义与概述 | Definition & Overview**
>
> *   **英文原文（源自PPT P31, P32）**：> American Fuzzy Lop (AFL) is a grey-box mutation-based feedback-based fuzzer. It is a well-known fuzzing tool for C/C++ code projects (and has been ported to other languages).> **Feedback-based**: It uses information from previous runs (like code coverage) to guide the selection of the next input.\
>     > **Mutation-based**: It changes (mutates) inputs to become new inputs.\
>     > **Grey-box**: It requires both seed inputs and code coverage information.
>
> *   **中文翻译**：> American Fuzzy Lop (AFL) 是一个基于灰盒、变异、反馈的模糊测试工具。它是针对 C/C++ 代码项目的著名模糊测试工具（并已被移植到其他语言）。> **基于反馈**：它利用前次运行的信息（如代码覆盖率）来指导下一次输入的选择。\
>     > **基于变异**：它通过改变（变异）输入来生成新输入。\
>     > **灰盒**：它需要种子输入和代码覆盖率信息。
>
> #### **📘 核心算法与策略 | Core Algorithm & Strategy**
>
> *   **英文原文（源自PPT P32, P33）**：> **Basic Strategy**:
>     >
>     > 1.  Start with a small set of input sample files as a queue of inputs.
>     > 2.  Mutate “randomly” an input file from the queue.
>     > 3.  If the mutated file reaches new or simpler path(s) (in log2 sense), add it to the queue.> **Mutation Operators** include:
>     >
>     > *   Deterministic: bit flips, byte flips, simple arithmetics (increment/decrement), overwriting with known interesting integers.
>     > *   Random (Havoc): stacked tweaks, clone/remove parts, splicing two inputs.
>
> *   **中文翻译**：> **基本策略**：
>     >
>     > 1.  以一小组输入样本文件作为输入队列开始。
>     > 2.  从队列中“随机”变异一个输入文件。
>     > 3.  如果变异后的文件覆盖了新的或更简单的路径（以 log2 意义衡量），则将其加入队列。> **变异操作符**包括：
>     >
>     > *   确定性：比特翻转、字节翻转、简单算术运算（递增/递减）、用已知的“有趣”整数覆盖。
>     > *   随机（混乱）：堆叠调整、克隆/删除部分、拼接两个输入。
>

> ## **🎯 详细说明：Delta Debugging**
>
> ### **📘 定义与动机 | Definition & Motivation**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（源自PPT P2, P12）**：> Delta Debugging is a method for **input reduction or code reduction**. It aims to remove some parts of a failure-inducing input to see if the reduced input still produces the same failure. If yes, treat the reduced input as a new failure-inducing input and continue the incremental removal.
>
> *   **中文翻译**：> Delta调试是一种**输入缩减或代码缩减**的方法。它旨在删除故障诱导输入的一些部分，以查看缩减后的输入是否仍能产生相同的故障。如果可以，则将缩减后的输入视为新的故障诱导输入，并继续增量删除。
>
> #### **📘 动机 | Motivation**
>
> *   **英文原文（源自PPT P3）**：> **Use case 1**: A web browser crashed when printing a webpage. If the webpage is large, it is a good idea to make the webpage smaller and at the same time able to crash the web browser before examining the details.> **Use case 2**: If the input data sequence is long (e.g., from UI event recorders like Appium), it would be better if we could shorten the input data sequence but reproduce these crashes.
>
> *   **中文翻译**：> **用例1**：网页浏览器在打印网页时崩溃。如果网页很大，最好能在检查细节之前将网页缩小，同时仍能使浏览器崩溃。> **用例2**：如果输入数据序列很长（例如来自Appium等UI事件记录器），最好能缩短输入数据序列但仍能重现这些崩溃。
>
> ### **📘 核心过程与策略 | Core Process & Strategy**
>
> #### **📘 过程 | Process**
>
> *   **英文原文（源自PPT P5, P6, P12）**：> **Inefficient strategy**: Remove single character one at a time (too fine-grained).\
>     > **Effective strategy**: First cut away large parts, then iteratively cut out the other parts bit by bit.> The process is iterative:
>     >
>     > 1.  Start with the original failure-inducing input.
>     > 2.  Divide the input into parts (e.g., halves).
>     > 3.  Test each part. If a part still causes the failure, use it as the new starting point.
>     > 4.  Repeat steps 2-3 until no further reduction is possible while still reproducing the failure.
>
> *   **中文翻译**：> **低效策略**：一次删除一个字符（粒度太细）。\
>     > **有效策略**：先切掉大块，然后迭代地一点一点切掉其他部分。> 该过程是迭代的：
>     >
>     > 1.  从原始故障诱导输入开始。
>     > 2.  将输入分成若干部分（例如两半）。
>     > 3.  测试每个部分。如果某部分仍能导致故障，则将其作为新的起点。
>     > 4.  重复步骤2-3，直到在仍能重现故障的前提下无法进一步缩减。
>
> #### **📘 示例 | Example**
>
> *   **英文原文（源自PPT P12, P13, P14）**：> **Example**: A buggy Quicksort program fails with input `[4, 12, 9, 14, 3, 10, 17, 11, 8, 7, 4, 1, 6, 19, 5, 21, 2, 3]` (output unsorted).
>     >
>     > *   Step 1: Cut the input in half, test first half `[4, 12, 9, 14, 3, 10, 17, 11, 8]`. It still fails.
>     > *   Step 2: Cut the failing half again. Test `[4, 12, 9, 14]` (passes), then test `[3, 10, 17, 11, 8]` (passes). Our guess wrong.
>     > *   Step 3: Try a different split of the last failing input (e.g., remove last two numbers). Test `[4, 12, 9, 14, 3, 10, 17]` (fails).
>     > *   ... Continue until a minimal failing input like `[4, 3, 2]` is found.
>
> *   **中文翻译**：> **示例**：一个有bug的快速排序程序对输入`[4, 12, 9, 14, 3, 10, 17, 11, 8, 7, 4, 1, 6, 19, 5, 21, 2, 3]`失败（输出未排序）。
>     >
>     > *   步骤1：将输入切成两半，测试前半部分`[4, 12, 9, 14, 3, 10, 17, 11, 8]`。仍然失败。
>     > *   步骤2：再次切分失败的那一半。测试`[4, 12, 9, 14]`（通过），然后测试`[3, 10, 17, 11, 8]`（通过）。我们的猜测错误。
>     > *   步骤3：尝试对上次失败的输入进行不同的分割（例如，删除最后两个数字）。测试`[4, 12, 9, 14, 3, 10, 17]`（失败）。
>     > *   …… 继续直到找到最小的故障输入，例如`[4, 3, 2]`。

### Testing Techniques

*   Oracles: Generic Regression, Metamorphic
*   Black-box: Equivalence Partitioning
*   White-box: Branch Coverage

> ## **测试预言 | Test Oracles**
>
> ### **📘 通用回归测试 | Generic Regression Testing**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（概念通用）**：> Generic Regression Testing is a testing approach that aims to ensure that previously developed and tested software still performs correctly after changes (such as enhancements, patches, or configuration changes). It typically involves re-running existing test cases (test suite) to detect any regressions (i.e., new bugs introduced by the changes).
>
> *   **中文翻译**：> 通用回归测试是一种测试方法，旨在确保先前开发和测试的软件在更改（如增强、补丁或配置更改）后仍能正确运行。它通常涉及重新运行现有的测试用例（测试套件）以检测任何回归（即由更改引入的新错误）。
>
> #### **📘 解释与示例 | Explanation & Example**
>
> *   **英文原文（概念通用）**：> In a continuous integration pipeline, regression tests are automatically run after each commit to prevent new code from breaking existing functionality. The oracle (expected outcome) for each test case is derived from the previously established correct behavior of the system.
>
> *   **中文翻译**：> 在持续集成流水线中，每次提交后会自动运行回归测试，以防止新代码破坏现有功能。每个测试用例的预言（预期结果）源自系统先前建立的正确行为。
>
> ### **📘 蜕变测试 | Metamorphic Testing**
>
> #### **📘 定义 | Definition**
>
> *   **英文原文（概念通用）**：> Metamorphic Testing is a property-based testing technique used when a precise oracle (expected output) is difficult or impossible to define. Instead of checking the exact output for a given input, it checks whether the program’s outputs satisfy certain metamorphic relations (properties) across multiple related inputs.
>
> *   **中文翻译**：> 蜕变测试是一种基于属性的测试技术，用于在难以或无法定义精确预言（预期输出）时使用。它不检查给定输入的精确输出，而是检查程序的输出在多个相关输入之间是否满足某些蜕变关系（属性）。
>
> #### **📘 解释与示例 | Explanation & Example**
>
> *   **英文原文（概念通用）**：> For example, in a sorting program, a metamorphic relation could be: if we reverse the input list, the output should be the reverse of the original sorted list. We test this relation without knowing the exact sorted list for every input.> Another example: in a sine function, sin(x) should equal sin(π - x). We can generate test cases based on this relation even if we don't have a table of exact sine values.
>
> *   **中文翻译**：> 例如，在排序程序中，一个蜕变关系可以是：如果我们反转输入列表，输出应该是原始排序列表的反转。我们无需知道每个输入的确切排序列表即可测试此关系。> 另一个例子：在正弦函数中，sin(x) 应等于 sin(π - x)。即使没有精确的正弦值表，我们也可以基于此关系生成测试用例。
>

>
> ## **六、黑盒测试：等价类划分 | Black-box Testing: Equivalence Partitioning**
>
> ### **📘 定义 | Definition**
>
> *   **英文原文（概念通用）**：> Equivalence Partitioning is a black-box testing technique that divides the input domain of a program into classes (partitions) of data from which test cases can be derived. The idea is that each partition should be treated equivalently by the program, so testing one representative value from each partition is sufficient.
>
> *   **中文翻译**：> 等价类划分是一种黑盒测试技术，将程序的输入域划分为数据的类（分区），从中可以导出测试用例。其思想是程序应等效地处理每个分区，因此测试每个分区的一个代表性值就足够了。
>
> ### **📘 解释与示例 | Explanation & Example**
>
> *   **英文原文（概念通用）**：> For example, consider a program that accepts an integer input from 1 to 100. Valid equivalence partitions could be: less than 1 (invalid), 1 to 100 (valid), and greater than 100 (invalid). We would select test cases like -5 (invalid), 50 (valid), and 150 (invalid).> The oracle for each test case is based on the expected behavior: invalid inputs should produce error messages or be rejected; valid inputs should be processed correctly.
>
> *   **中文翻译**：> 例如，考虑一个接受 1 到 100 整数输入的程序。有效的等价类划分可以是：小于1（无效）、1到100（有效）、大于100（无效）。我们会选择像 -5（无效）、50（有效）、150（无效）这样的测试用例。> 每个测试用例的预言基于预期行为：无效输入应产生错误信息或被拒绝；有效输入应被正确处理。

> ## **七、白盒测试：分支覆盖 | White-box Testing: Branch Coverage**
>
> ### **📘 定义 | Definition**
>
> *   **英文原文（概念通用）**：> Branch Coverage (also known as Decision Coverage) is a white-box testing metric that measures the percentage of branches (decision points) in the code that have been executed during testing. A branch is typically an outcome of a conditional statement (if, switch, loop condition). 100% branch coverage means every possible branch (both true and false outcomes) has been taken at least once.
>
> *   **中文翻译**：> 分支覆盖（也称为决策覆盖）是一种白盒测试度量标准，用于衡量在测试期间已执行的代码中分支（决策点）的百分比。分支通常是条件语句（if、switch、循环条件）的结果。100% 分支覆盖意味着每个可能的分支（真和假结果）都至少被执行过一次。
>
> ### **📘 解释与示例 | Explanation & Example**
>
> *   **英文原文（概念通用）**：> For example, consider the code: `if (x > 0) { y = 1; } else { y = -1; }`. To achieve 100% branch coverage, we need two test cases: one with x > 0 (to take the true branch) and one with x <= 0 (to take the false branch).> Branch coverage is stronger than statement coverage because it ensures that both outcomes of decisions are tested, revealing bugs that might hide in untaken branches.
>
> *   **中文翻译**：> 例如，考虑代码：`if (x > 0) { y = 1; } else { y = -1; }`。要实现 100% 分支覆盖，我们需要两个测试用例：一个 x > 0（执行真分支）和一个 x <= 0（执行假分支）。> 分支覆盖比语句覆盖更强，因为它确保测试决策的两种结果，从而揭示可能隐藏在被未执行分支中的错误。
