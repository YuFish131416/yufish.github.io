## 一、VR 系统架构（VR System Architecture）

### 1.1 VR 系统的四大核心任务

1.  **读取输入设备**（Read Input Devices）：从传感器获取用户输入
2.  **更新虚拟世界状态**（Update Virtual World State）：根据输入更新场景数据库
3.  **渲染输出**（Render Outputs）：生成图形、触觉等各类感知反馈
4.  **输出到外设**（Feed Output Devices）：将渲染结果发送至显示器、触觉设备等

### 1.2 VR 引擎架构需求

VR 引擎需要强大的计算机架构，以实现：

*   **低计算延迟**（Low Latency）
*   **快速图形渲染**（Fast Graphics Rendering）
*   **触觉渲染**（Haptics Rendering）

***

## 二、图形渲染管线（Graphics Rendering Pipeline）

**渲染（Rendering）**：将虚拟世界中的三维几何模型转换为二维图像呈现给用户的过程。

**管线化（Pipelining）**：将渲染过程划分为多个阶段，并将各阶段分配给不同的硬件资源并行执行。

### 2.1 管线的三个阶段

| 阶段 | 名称                          | 运行位置 |
| -- | --------------------------- | ---- |
| 1  | **应用阶段（Application Stage）** | CPU  |
| 2  | **几何阶段（Geometry Stage）**    | GPU  |
| 3  | **光栅化阶段（Rasterizer Stage）** | GPU  |


***

### 2.2 应用阶段（Application Stage）

*   由 CPU **软件执行**

*   管理虚拟世界的**几何数据库（Geometry Database）**

*   循环执行以下任务：

    *   获取用户输入（鼠标、追踪器、感知手套等）
    *   更新视角、数据库与虚拟世界状态
    *   选择相关几何对象并发送至后续阶段进行渲染

***

### 2.3 几何阶段（Geometry Stage）

*   通常**在硬件（GPU）中实现**

*   接收应用阶段产生的**几何图元列表**进行处理

*   主要任务：

    *   **模型变换（Model Transformations）**
    *   **着色计算（Shading Computations）\\**
    *   **场景投影（Scene Projection）**
    *   **裁剪（Clipping）**

#### Phong 光照模型（Phong Illumination Model）

$$
I_\lambda = I_{a\lambda} K_a C_{d\lambda} + f_{att} I_{p\lambda} (K_d C_{d\lambda} \cos\theta + K_s C_{s\lambda} \cos^n\alpha) 

$$

| 符号                               | 含义                            |
| -------------------------------- | ----------------------------- |
| $ I_{a\lambda} K_a C_{d\lambda}$ | **环境光反射（Ambient Reflection）** |
| $K_d C_{d\lambda} \cos\theta$    | **漫反射（Diffuse Reflection）**   |
| $K_s C_{s\lambda} \cos^n\alpha$  | **镜面反射（Specular Reflection）** |
| $ f_{att} $                      | 大气衰减因子（如雾、烟雾效果）               |


> ⚠️ **考点提示**：光源数量越多，$I_\lambda$ 的计算代价越大，可能成为瓶颈。部分 GPU 会限制最大光源数量。

#### 多边形着色模型（Polygon Shading Models）

| 模型                  | 描述                | 特点           |
| ------------------- | ----------------- | ------------ |
| **Flat Shading**    | 每个多边形所有像素颜色相同     | 计算最快，质量最低    |
| **Gouraud Shading** | 基于顶点颜色对多边形内部进行插值  | 中等质量         |
| **Phong Shading**   | 对顶点法线插值后逐像素计算光照强度 | **质量最高**，最真实 |


***

### 2.4 光栅化阶段（Rasterizer Stage）

*   通常**在硬件（GPU）中实现**

*   将几何阶段输出的顶点信息（x, y, z, color, texture）转换为像素信息

*   主要任务：

    *   将几何图元（三角形、矩形）分解为**像素片段（Pixel Fragments）**
    *   **反锯齿（Antialiasing）**
    *   **纹理映射（Texture Mapping）**

#### 反锯齿（Antialiasing）工作原理

*   将每个像素细分为  $n$  个**子像素（Subpixels / Sub-samples）**

*   对每个子像素计算颜色

*   最终颜色 = 所有子像素颜色的**平均值**

*   子像素数量越多，反锯齿效果越好，但计算代价也越大

#### 双缓冲（Double Buffering）

| 缓冲区                   | 作用             |
| --------------------- | -------------- |
| **前缓冲（Front Buffer）** | 存储当前正在显示的上一帧图像 |
| **后缓冲（Back Buffer）**  | 存储当前正在渲染的新图像   |


*   渲染完成后将后缓冲与前缓冲**交换（Swap）**，新图像立即显示
*   **作用**：消除画面闪烁（Flicker）

***

### 2.5 管线瓶颈检测与优化（Pipeline Bottlenecks）

#### 瓶颈类型识别

| 瓶颈类型                  | 判断条件              | 所在阶段  |
| --------------------- | ----------------- | ----- |
| **CPU-limited**       | CPU 使用率达 100%     | 应用阶段  |
| **Transform-limited** | 减少光源数或多边形数后性能提升   | 几何阶段  |
| **Fill-limited**      | 降低图像分辨率或多边形数后性能提升 | 光栅化阶段 |


#### 各阶段优化方法

**应用阶段优化：**

*   更换更快的 CPU（更高主频 / 更多核心）
*   使用更快的内存（RAM）
*   使用更优化的编译器
*   优化代码（降低计算精度、减少除法运算次数）

**几何阶段优化：**

*   减少光源数量
*   减少场景复杂度（即减少多边形数）

**光栅化阶段优化：**

*   降低图像分辨率（减少需要处理的像素数）
*   减少多边形数（与几何阶段共通）

## 三、触觉渲染管线（Haptics Rendering Pipeline）

触觉管线包含三个阶段：

```
用户输入
  → 碰撞检测阶段（Collision Detection Stage）
  → 力计算阶段（Force Computation Stage）
  → 触觉计算阶段（Tactile Computation Stage）
  → 触觉显示设备（Haptic Interface）
```

### 3.1 碰撞检测阶段（Collision Detection Stage）

*   加载对象的**物理特性**（表面光滑度、表面温度、重量等）
*   执行**碰撞检测**，判断用户与对象之间是否发生碰撞
*   将碰撞结构传递到下一阶段

### 3.2 力计算阶段（Force Computation Stage）

*   **计算力（Computing Force）**：基于物理仿真模型计算碰撞力

    *   碰撞对象越多、力计算越复杂 → 该阶段越可能成为瓶颈（**force-limited**）

*   **力平滑（Force Smoothing）**：调整力向量方向，避免突变

*   **力映射（Force Mapping）**：将计算出的力投影到特定触觉设备的特性范围内

### 3.3 触觉计算阶段（Tactile Computation Stage）——触觉纹理

*   渲染仿真的**触觉反馈分量**
*   将计算出的效果（振动、表面温度等）叠加到力向量上，发送至触觉显示设备

## 四、GPU 架构（GPU Architecture）

### 4.1 GPU vs CPU 对比

| 特性       | CPU                  | GPU                    |
| -------- | -------------------- | ---------------------- |
| **编程模型** | 标量，无原生数据并行           | 大规模数据并行                |
| **擅长领域** | 控制密集型任务              | 数据密集型任务                |
| **算术单元** | 少，但速度快               | 多，专为并行计算               |
| **优化目标** | 低延迟                  | 高带宽                    |
| **增长速率** | ~1.5× / 年（~60× / 十年） | ~2.0× / 年（~1000× / 十年） |


### 4.2 图形渲染的特点（决定 GPU 设计方向）

*   大量数据需要处理
*   大量并行计算
*   简单的控制结构
*   多阶段前馈管线
*   延迟容忍/深层管线

> 因此，GPU 芯片空间大量用于**算术单元**，而非分支控制电路。

### 4.3 硬件图形管线

**传统管线：**

```
Application → Transform → Rasterizer → Shade → Video Memory (Textures)
    CPU    |              GPU                              GPU
```

**高级管线（含可编程处理器）：**

```
Application → Vertex Processor → Rasterizer → Fragment Processor → Video Memory
    CPU    |         GPU                              GPU
```

| 处理器                                | 主要任务                    |
| ---------------------------------- | ----------------------- |
| **Vertex Processor**（顶点处理器，并行多个）   | 坐标变换、逐顶点光照计算、背面剔除、裁剪    |
| **Rasterizer**（光栅化器）               | 将顶点转换为像素片段；对顶点属性进行像素间插值 |
| **Fragment Processor**（片段处理器，并行多个） | 深度比较、逐像素颜色计算、纹理采样       |


### 4.4 GPU 内存寻址

*   当前 GPU **不支持直接内存寻址**
*   仅支持**向量寻址**（1D、2D、3D 等）
*   原因：简单的内存地址方案有助于高效地通过管线传输大量数据

### 4.5 现代 GPU 特性

*   **高度可编程**：可编程的顶点处理器与片段处理器，支持高级语言编程
*   **高精度计算**：全管线支持 32-bit 浮点（单精度）
*   **通用计算**：GPU 强大的计算能力使其成为深度学习等通用计算的理想平台（如 NVIDIA CUDA）

## 五、分布式 VR 架构（Distributed VR Architecture）

**分布式 VR 引擎**：使用**两个或更多渲染管线**的 VR 系统，可位于单台计算机、多台局域网计算机或多台远程计算机上。

### 5.1 系统分类

| 类别        | 子类型                                                 |
| --------- | --------------------------------------------------- |
| **单用户系统** | 多并排显示器；多台局域网计算机                                     |
| **多用户系统** | 客户端-服务器（Client-Server）；点对点（Peer-to-Peer）；混合（Hybrid） |


### 5.2 多管线同步（Multi-pipeline Synchronization）

无论多个图形管线位于同一台还是多台计算机，输出图像都必须**同步**，以：

*   减少整体系统延迟
*   保持帧率一致性

#### 三种同步方法

| 方法                                      | 机制                              | 局限性                |
| --------------------------------------- | ------------------------------- | ------------------ |
| **软件同步（Software Synchronization）**      | 同步各管线的应用阶段，使其同时开始处理新帧           | 未考虑各管线负载不对称的情况     |
| **帧缓冲同步（Frame Buffer Synchronization）** | 同步缓冲区交换，等待显示器刷新信号后再交换           | 水平/垂直刷新未同步         |
| **视频同步（Video Synchronization）**         | 一个显示器为主，其余为从；从设备的内部视频逻辑电路与主设备同步 | 最佳方案是软件+帧缓冲+视频三重同步 |


> ⚠️ **考点提示**：**最佳方法**是软件 + 帧缓冲 + 视频三重同步。

### 5.3 图形与触觉管线的同步

*   同步在**应用阶段**完成

*   两种实现方案：力计算在主机 CPU 执行；或由触觉接口控制器内置处理器执行

*   **必须解耦**两个管线，因为二者输出频率差异巨大：

    *   PHANToM 触觉控制器：**1 KHz**（1000次/秒）
    *   图形管线：**30–60 fps**

## 六、PC 集群（PC Clusters）

### 6.1 为何需要 PC 集群

大型拼接显示（Tiled Displays）由数十台投影仪组成，若需高分辨率，则每台投影仪需要一条独立图形管线。单台 PC 无法满足，原因：

*   **PCIe 插槽数量不足**
*   **总线拥堵**导致吞吐量下降

### 6.2 两种替代方案对比

| 方案                       | 优点               | 缺点                            |
| ------------------------ | ---------------- | ----------------------------- |
| **大型多管线工作站**（如 PanaWall） | 专用硬件，性能稳定        | 成本极高；每条管线仅支持 1–2 个显示器         |
| **PC 集群**                | 成本显著降低；图像拼接由软件完成 | 需要高速 LAN；LAN 吞吐量成为集群规模的主要限制因素 |


### 6.3 PC 集群架构要点

*   PC 集群必须通过**高速局域网**连接，以便控制服务器同步各渲染节点的输出图形
*   **固定的 LAN 吞吐量**是集群规模的主要限制因素

### 6.4 典型案例

**Princeton Display Wall：**

*   24 台 PC 投影仪拼接，显示分辨率：**6000 × 3000** 像素
*   投影面积：18′ × 8′ 后投影屏幕
*   网络：**100Mb 以太网**

**Chromium PC 集群（Stanford University）：**

*   32 台渲染服务器 + 4 台控制服务器（各配 NVIDIA 显卡）
*   网络：**64-bit 66MHz LAN**，点对点带宽 **100 MB/s**
*   软件：**开源**（SourceForge）

### 6.5 WireGL 软件

*   专为集群拼接渲染设计的可扩展软件系统
*   拦截 VR 仿真程序发出的 **OpenGL 命令**，编码后分发至对应渲染服务器
*   **数据只发送给需要的 PC**：几何数据按屏幕空间排序，每台服务器只接收与其显示区域重叠的几何体
*   每台渲染服务器维护独立的**网络缓冲区**
*   仅在渲染服务器状态与网络缓冲区存在差异时才传输数据（节省带宽）

**WireGL 性能：** 相比广播式网络，WireGL 随集群规模扩大（投影仪增多）帧率下降远更平缓，具有更好的可扩展性。
